# 玩转通用智能体之Claude Code

## 第二十章 常见失败模式与对策

我年轻时做很多事情都会失败。写代码有bug，做项目延期，跟人沟通产生误会。后来我学会了一件事：失败不可怕，重要的是从失败中学习，避免下次再犯。

用Claude Code也会失败。以下是一些常见的失败模式，以及应对的办法。

### 一

**失败模式1：Context溢出**

症状：聊了很久之后，Claude Code开始胡说八道，或者忘记之前的约定。

原因：Context window满了，旧的内容被挤掉了。

对策：及时重启会话，或者使用`/compact`清理。复杂任务拆成子代理处理。

### 二

**失败模式2：过度修改**

症状：你让它改一个小地方，结果它把整个文件甚至整个项目都重构了。

原因：你的指令不够具体，或者没有限定范围。

对策：明确说"只改这个函数"、"不要动其他文件"。用Plan模式先让它列出要改什么，你确认后再执行。

### 三

**失败模式3：理解偏差**

症状：Claude Code做出来的东西跟你想要的不一样。

原因：你们对同一个问题的理解不同。

对策：多用例子说明。比如"按照这种风格"、"参考这个文件的做法"。做完后及时检查，有问题立即纠正。

### 四

**失败模式4：循环修正**

症状：让它改A，结果B坏了；修好B，A又出问题了。来回折腾好几轮。

原因：Claude Code没有全局视野，修一个点可能影响到其他点。

对策：让它在修改前分析影响范围。要求它跑测试验证。如果反复出问题，考虑人工介入。

### 五

**失败模式5：安全失误**

症状：不小心删了重要文件，或者泄露了敏感信息。

原因：没有仔细确认Claude Code的操作，或者把机密信息暴露给它。

对策：敏感操作一定要看清楚再确认。不要把密码、密钥写在代码里。定期备份重要数据。

### 六

失败是正常的，每个人都会遇到。

关键是要有应对的预案：
- 版本控制（Git）是救命稻草，出问题了可以回滚
- 定期备份，不要把鸡蛋放在一个篮子里
- 重要操作人工复核，不要完全依赖AI

Claude Code是个好工具，但它不是万能的，也会犯错。把它当成一个能力很强的助手，而不是全知全能的神。保持警惕，保持控制，才能安全地使用它。
