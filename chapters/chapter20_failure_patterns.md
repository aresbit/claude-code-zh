# 玩转通用智能体之Claude Code

## 第二十章 常见的翻车与对策

我年轻时做事常翻车：写代码有 bug，做项目误期，跟人沟通生误会。后来学会一桩事：翻车不可怕，要紧的是从里头学点东西，下回别再栽在同一处。

用 Claude Code 也会翻车。下面是几样常见的翻法，连带对策。

### 一

**翻法一：上下文溢出**

症候：聊了老半天，它开始胡咧咧，或者把先前的约定忘了。

根由：上下文窗口满了，旧的内容被挤了出去。

对策：趁早清场重来，或者使 `/compact` 理一理。活儿太大，就拆给子代理。

### 二

**翻法二：改过了头**

症候：你让它动一个小地方，它把整个文件、乃至整个项目都翻新了。

根由：你的吩咐不够具体，或者没把范围圈死。

对策：明说「只改这个函数」「别动别的文件」。先用计划模式让它把要改的列出来，你点头了再放它执行。

### 三

**翻法三：领岔了意思**

症候：它做出来的东西，跟你想要的不是一回事。

根由：你俩对同一件事的理解，根本不在一处。

对策：多拿例子说话——「照这个风格」「参着这个文件的做法」。做完趁早查验，不对立马拨正。

### 四

**翻法四：来回拆补**

症候：让它改甲，乙坏了；修好乙，甲又出毛病。来回折腾好几趟。

根由：它没有全局的眼光，补一处可能戳漏另一处。

对策：让它动手前先把牵连的范围摸一遍，要它跑测试验一验。要是反复出事，就该人工插手了。

### 五

**翻法五：安全失手**

症候：失手删了要紧文件，或者把机密给抖搂了出去。

根由：没看仔细就点了头，或者把机密信息亮给了它。

对策：要紧的动作一定看清楚再应。口令、密钥别往代码里写。要紧的数据，时常备份。

### 六

翻车是常事，谁都免不了。

要紧的是手里得有应急的预案：

- 版本控制（Git）是救命的稻草，出了事还能回退
- 时常备份，别把鸡蛋搁一个篮子里
- 要紧的动作人工过一遍目，别把命全交给 AI

Claude Code 是件好家伙，可它不是万能的，也会出错。把它当一个能耐很大的帮手，别当一个无所不能的神。留着戒心，攥着缰绳，才使得安稳。
