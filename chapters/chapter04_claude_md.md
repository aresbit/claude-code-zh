# 玩转通用智能体之Claude Code

## 第四章 CLAUDE.md：与智能怪兽的契约

用了 Claude Code 一阵子，我发觉一件烦心事：每开一个新会话，都得把些老底重新交代一遍——我使的什么框架，我的代码是什么脾气，有哪些忌讳。

这就好比每天早上一睁眼，都得跟室友重新认识一回：「我叫王小二，爱喝豆浆，别动我的牙刷。」烦不烦。

### 一

CLAUDE.md 就是为了了断这桩烦心事来的。

这文件搁在项目根目录，或者你家目录下的 `.claude/` 里。Claude Code 每回启动都先读它，把里头的话当作底细记下。

你可以往里写：

- 项目是干什么的
- 用了哪些技术
- 代码风格的约定
- 常用的命令
- 该留神的地方

一句话，凡是你想让它心里有数的事，都能写进去。

### 二

我写 CLAUDE.md 的光景，有点像写遗书。

起初不知写什么好，只草草记了几条：项目用什么语言、什么框架、测试怎么跑。后来才品出来，这文件还能写得更细些。

譬如我添了一条：「别用 `any` 类型，再麻烦也得把类型定义写对。」这是因为 Claude Code 偶尔偷懒，觉着「能跑就成」。可我有洁癖，见不得 `any`。

我还写了：「写注释，说清楚为什么这么做，别光说做了什么。做了什么，看代码便知；为什么要做，才值得费笔墨。」

这些约定，让它的出活更对我的胃口。

### 三

不过 CLAUDE.md 也不是越长越好。

前头提过那个「上下文窗口」，它是有数的。你写得太满，就挤占了 Claude Code 思量的地方。好比给一个人塞一摞背景材料，他反倒抓不住重点。

所以 CLAUDE.md 得写得精。只写真要紧的，写它自个儿猜不着的。代码能讲明白的，就不必写；标准库怎么用，也不必写——这些它门儿清。

Anthropic 官方给了个单子：

该写的：

- 特殊的 bash 命令
- 跟默认不一样的代码风格
- 测试的法子
- 项目的惯例（分支怎么命名、PR 有什么规矩）
- 架构上的取舍
- 开发环境的特殊设置
- 容易踩的坑

不该写的：

- 看代码就知道的事
- 语言的标准规范
- 长篇的 API 文档（贴个链接就够）
- 老变的信息
- 每个文件的细枝末节
- 明摆着的道理（比如「要写干净的代码」）

### 四

我拿 `/init` 生出来的那份 CLAUDE.md，后来改了一遍又一遍。

每回 Claude Code 做了什么让我不顺眼的事，我就琢磨：这是不是该在 CLAUDE.md 里提前打个招呼？是，就添进去。

就这么着，这文件越来越长，也越来越合我的意。

像是调收音机。起头满是杂音，你一点点拧，直到拧出最清亮的那个频道。

### 五

CLAUDE.md 是人和怪兽之间的字据。

你立下规矩，它守着规矩。可这字据不是一立永逸的，你得时常改它、补它，让它更趁手。

记着一条：CLAUDE.md 是给眼下这个项目用的。项目不同，CLAUDE.md 也该不同。你个人的偏好，写进个人设置；项目的特殊讲究，写进项目里那份 CLAUDE.md。

如此一来，不管你手上是什么活，Claude Code 都知道该怎么搭你的腔。
