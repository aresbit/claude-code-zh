# 玩转通用智能体之Claude Code

## 第五章 Plan 模式：三思而后行

我年轻时有个毛病：动手之前不肯先想清楚，干了再说。这毛病害我吃过不少亏——墙上钉钉子，钉穿了电线；写代码，写到半截发觉路子错了，又得推倒重来。

上了点年纪才学会一桩事：慢，就是快。

### 一

Claude Code 有个计划模式（Plan mode），就是逼你「慢下来」的物件。

平日里你跟它说话，它张口就干：让它改代码，它就改；让它跑命令，它就跑。这固然快，可有时也坏事。

计划模式不一样。按 `Shift+Tab` 在几种模式间切换，切到计划模式，再把你要做的事讲给它听——它会先拟个方案给你看，等你点头，才真动手。

这就像请了支装修队。寻常你一说「把墙刷成蓝的」，他们抄起刷子就上；计划模式下，他们先把效果图、报价单、施工的章程摆到你跟前，你应了，才开工。

### 二

我用计划模式最勤的时候，是动大手术之前。

譬如要重构一个模块。寻常模式下撂一句「重构这个模块」，它没准儿立时就改，改了一摞文件，末了发觉方向不对。可计划模式下，它会先把现状捋一遍，提个重构的章程，列清要动哪些文件，每一步打算做什么。

我便能把这章程过一遍目：提点意见，或者干脆驳回重来。等章程定了，再放它去执行。

这让我想起一句老话：谋定而后动。

### 三

计划模式还有个好处：让你把它的「心思」看得更真切。

有时候你会撞见，它把你的意思领岔了。你说的「重构」是这个意思，它却听成了那个意思。在计划模式下，这点偏差能早早现形，及时拨正；搁寻常模式，等你瞧出不对，它怕是已经改花了一片文件。

这就跟跟人搭伙一样：默契还没攒够的时候，最好把话讲透，确认对方领会对了，再一道动手。

### 四

可计划模式也是有代价的——慢。

每回都得等它拟章程，你过目，兴许还要回炉，然后才执行。碰上鸡毛蒜皮的小改动，这就有点杀鸡用牛刀了。

所以我的经验是：小改动走寻常模式，大改动走计划模式。什么算大？要动好几个文件的，会牵连别处功能的，可能引出破坏性变更（breaking changes）的——这些当口，用计划模式稳妥。

### 五

还有一手，是把耗时的长活儿丢到后台去跑。

譬如跑整套测试、料理一大堆文件，犯不上你干瞪眼守着。撂给它在后台慢慢做，你抽身去忙别的，等它做完了再回头招呼你。

像是手底下有个帮手：事交代下去，自己先去喝杯茶，做妥了他来跟你回话。

说到底，计划模式也好，后台任务也好，都是让你把这只怪兽攥得更稳的家什。什么场合使什么模式，拿捏到位，便是驭兽的高招了。
