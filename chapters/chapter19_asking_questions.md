# 玩转通用智能体之Claude Code

## 第十九章 提问的艺术：好问题带来好答案

我年轻时有个老师，特别会提问。同样的知识点，他问出来的问题总是特别有意思，让人忍不住想思考。后来我发现，提问是一门艺术，好问题能引出好答案，坏问题只能得到敷衍的回复。

用Claude Code也是一样。你怎么问，决定了它怎么答。

### 一

什么是好问题？

首先，要明确。不要说"这个不好，改一下"，太模糊。是说功能有问题？性能有问题？还是有bug？

好的问题是："这个函数在处理空数组时会崩溃，帮我加个检查"

其次，要有上下文。不要假设Claude Code知道你在想什么。把相关的背景信息给它：

"我在做一个用户管理系统，用的是Node.js和MongoDB。现在有个问题：查询用户列表时，如果数据量太大，响应会很慢。"

### 二

还有，要分步骤。

复杂的问题，不要期望一步到位。拆成几个小问题，一个个解决。

比如你要重构一个模块，可以这样：

1. "先分析一下这个模块目前的结构，有什么主要问题？"
2. "针对这些问题，你有什么重构建议？"
3. "就按第二个建议来做，先改接口定义"
4. "现在改实现部分"
5. "最后加一下测试"

这样一步步来，比直接说"重构这个模块"要可控得多。

### 三

学会追问也很重要。

Claude Code的回答，有时候不够深入，或者没有完全回答你的问题。这时候要继续追问：

- "能不能详细解释一下这一点？"
- "还有其他的方案吗？"
- "这样做有什么风险？"

把问题挖深，才能得到更有价值的信息。

### 四

但也要注意，不要问太多。

Context window有限，问题太长了，之前的对话就被挤掉了。而且问题太多，Claude Code可能会漏掉一些。

我的经验是，一次问3-5个相关问题，等回答完了再问下一批。这样既不会遗漏，又不会超出限制。

### 五

最后，要学会验证。

Claude Code给的建议，不一定都是对的。特别是涉及技术细节的时候，它可能会出错。

所以重要的信息要验证：
- 让它给出来源
- 自己查文档确认
- 小范围测试一下

这就像医生开药，药师要审核，患者也要看看说明书，不能盲目相信。

总之，提问是门艺术。问题问得好，Claude Code就能发挥出最大的价值。多练习，多总结，你会越来越擅长跟这只智能怪兽对话。
