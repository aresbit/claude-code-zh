# 玩转通用智能体之Claude Code

## 第九章 Subagents：分身有术

小时候看《西游记》，最羡慕孙悟空的毫毛分身。拔一把毫毛，吹口气，就变出几百个小猴子，一起上阵打仗。我想，要是我也能有这本事该多好，一个我去上学，一个我去打球，还有一个我在家睡觉。

后来长大了，知道分身是假的了。但用Claude Code的时候，我发现了一件类似的事情：Subagents。

### 一

Subagents是什么？

简单说，就是你可以让Claude Code创建子代理，每个子代理处理一部分任务，然后汇总结果。这就像是一个项目经理，把大任务拆成小任务，分给不同的组员，最后整合大家的成果。

为什么要这样做？

因为context window的限制。前面说过，Claude Code的记性有限，任务太复杂它就hold不住。但如果拆成多个子任务，每个子任务给不同的子代理，就能绕过这个限制。

### 二

怎么创建Subagent？

用`/agent`命令，或者在一些操作中选择使用子代理。比如你可以说：

```
分析一下这个项目的架构，用多个子代理并行处理不同模块。
```

Claude Code就会自动创建多个Subagents，一个负责前端，一个负责后端，一个负责数据库，等等。它们各自分析自己负责的模块，最后汇总成一个完整的架构图。

### 三

Subagents还有一个用处，就是隔离风险。

有时候你想让Claude Code做一些可能有风险的操作，比如批量修改很多文件。如果让它在主会话里做，万一搞砸了，可能会污染整个会话的上下文。

但如果创建一个Subagent，让它在隔离的环境里做这件事，就算搞砸了，也只会影响那个子代理，主会话还是干净的。

这就像做化学实验，有些危险的反应要在通风橱里做，不能把整个实验室都搞脏了。

### 四

但Subagents也不是万能的。

首先是协调成本。多个子代理并行工作，最后需要整合结果。这个整合的过程有时候挺麻烦的，特别是当各个子代理的结果有冲突的时候。

其次是资源消耗。每个子代理都要占用一定的计算资源，子代理太多了，可能会变慢，甚至超出使用限额。

所以用Subagents要适度。小任务没必要拆分，大任务才需要。一般来说，当你发现主会话开始迷糊了，或者任务涉及太多文件了，就该考虑用Subagents了。

### 五

总之，Subagents是Claude Code应对复杂任务的重要手段。它让你能像孙悟空一样分身有术，把大问题拆成小问题，各个击破。记住一点：合理拆分，适度使用，才能发挥它的最大威力。
