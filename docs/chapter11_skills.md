## 第十一章 Skills：打一件趁手的兵器

我年轻时玩过一款游戏，里头兵器五花八门——剑、刀、枪、戟，各有各的路数。后来出了个锻造的玩法，你能自个儿打兵器，按自己的心意定属性。那阵子我觉着这事真带劲。

Claude Code 的 Skills（技能），就有这么点意思。

### 一

Skills 是什么？

说白了，就是你能给 Claude Code 定制一身专门的本事和门道。你写一个技能文件，讲清在某种场合它该怎么使唤这身本事，往后到了那个场合，它就自个儿把这技能取出来用。

打个比方：CLAUDE.md 像一本通用的员工手册，把公司的基本规矩告诉它；Skills 则像专业培训，教它怎么对付某一类活计。

### 二

技能文件搁在哪儿？

有三处：

1. 项目级：`.claude/skills/<技能名>/SKILL.md`
   - 只对眼下这个项目管用

2. 个人级：`~/.claude/skills/<技能名>/SKILL.md`
   - 对你名下所有项目都管用

3. 企业级：在管理后台里配
   - 对整个组织的人都管用

技能文件长什么样？大致是这副骨架：

```markdown
---
name: code-review
description: 做代码审查，挑出潜在的毛病和可改进处
---

做代码审查时，请照下面的步子来：

1. **安全**
   - 看有没有 SQL 注入的口子
   - 看有没有 XSS 的窟窿
   - 看有没有敏感信息外漏

2. **性能**
   - 看有没有 N+1 查询
   - 看有没有多余的循环
   - 看内存使唤得怎样

3. **可读**
   - 函数是不是太长
   - 变量起名清不清楚
   - 该有的注释有没有

出活的格式：
- 要命的毛病（非改不可）
- 改进的建议（可选）
- 夸一夸（做得好的地方）
```

头上那段 YAML 交代技能的底细，底下那段 Markdown 才是具体的吩咐。

### 三

最妙的是，写好的技能你还能 `/技能名` 这样直接喊它出来，也可以撂着不管——它瞧出眼下这活儿对路，会自个儿取来用。这就跟兵器架上的家伙一样：趁手的，随手就抄；摆着的，到了用时它自会想起。

### 四

Skills 的好处在哪？

头一桩是省着重来。一个技能写好了，多个项目都使得，不必每回从头交代。

第二桩是齐整。一队人都用同一个技能，代码审查也好、写文档也好、生成测试也好，出来的路数是一致的。

第三桩是专。你能为某种技术栈专写一个技能——React 的、Rust 的、Kubernetes 的——往后它对付相干的代码，就更在行。

### 五

可技能也不是越多越好。

技能一多，加载就慢；技能之间要是打起架来，还会闹出些古怪的举动。

所以我的主张是：

- 只留常使、顶用的
- 时常清一清，把不用的删了
- 留意技能彼此搭不搭

这就像工具箱：家伙得有，可别塞得太满，塞满了反倒不好找。

说到底，Skills 是定制 Claude Code 的一手要紧本事。把技能写好，就像给自己打一件趁手的兵器，能让这只怪兽更称你的心。
