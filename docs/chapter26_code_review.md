## 第二十六章 自动代码审查：一群怪兽替你挑刺

第十七章讲过 GitHub Actions，那是把怪兽塞进自己的 CI 流水线。这一章讲一个更省事的东西：**Code Review**——Anthropic 托管的自动 PR 审查，装好就用，不用你自己搭流水线。

### 一

它干的活是这样的：你开了个 PR，它派**一队各管一摊的智能体**，结合你**整个代码库**的上下文，去看这次改动——找逻辑错误、安全漏洞、没考虑到的边界、藏得很深的回归。找到了，就**直接评论在出问题的那几行代码上**。

关键是它**不批准、也不卡你的 PR**。findings 按严重程度打标签，你原来的审查流程一点不动：

| 标记 | 程度 | 含义 |
|---|---|---|
| 🔴 | Important | 合并前该修的真 bug |
| 🟡 | Nit | 小毛病，值得修但不挡路 |
| 🟣 | Pre-existing | 代码库里本来就有、不是这个 PR 引入的 |

每次审查平均跑 20 分钟，按 token 计费，一次大约 $15–25，随 PR 大小和复杂度浮动。（这功能也是 research preview，面向 Team 和 Enterprise；开了零数据保留 ZDR 的组织用不了。）

### 二

装它得管理员出马。去 `claude.ai/admin-settings/claude-code` 找到 Code Review，点 Setup，装上 Claude 的 GitHub App，再挑哪些仓库启用。每个仓库可以单独设触发时机：

- **PR 创建后审一次**
- **每次 push 都审**（最勤快，也最烧钱）
- **手动**（只有人在 PR 上喊了才审）

### 三

手动喊它，两条命令，发在 PR 的**顶层评论**里（不是某行代码上的内联评论）：

- `@claude review`：审一次，并且**订阅**这个 PR——往后每次 push 都审。
- `@claude review once`：只审这一次，不订阅。

长命 PR、改来改去 push 很勤的，用 `once` 那条，省得每 push 一次烧一次钱。

### 四

它默认只盯**正确性**——会把生产搞挂的 bug，而不是格式、缩进这类口味问题。想调教它，仓库里放两个文件：

- **`CLAUDE.md`**：你项目通用的规矩。它读，并把新引入的违反当 nit 级提出来。
- **`REVIEW.md`**：**只管审查**的指令，优先级最高，直接灌进审查流水线里每个智能体的脑子。

`REVIEW.md` 能干的事不少：重新定义对你这个仓库来说什么才算 🔴 Important、给 nit 数量封顶（「最多报五条，剩下的在总结里提一句数量」）、跳过生成代码和 lockfile、加上「新 API 路由必须有集成测试」这类硬规矩。

我的经验是：`REVIEW.md` 别写太长。写长了，真正要紧的规矩反而被稀释。

### 五

不想装 GitHub App，就想在本地终端先审一遍再推？有个命令：

```text
/code-review
```

在任何会话里跑，默认审你这个分支领先上游的提交、加上工作区里没提交的改动。`--comment` 把 findings 发成 PR 内联评论，`--fix` 直接把修改应用到工作区。

（小历史：这命令在 v2.1.147 之前叫 `/simplify`。如今 `/simplify` 改成了只做清理、不找 bug 的那一档；你要找 bug，用 `/code-review`。）

### 六

我在机关待过，知道「审材料」这活儿有多磨人——一份文件来回看，错别字、口径、数据，一项项抠，眼都看花。最怕的是看走眼，一个数错了，盖了章发出去，谁也担待不起。

这一队怪兽的好处，是它们不眨眼、不犯困、还各管一摊互相验。但你也别全指望它——它给的是「参考意见」，红黄紫标得清清楚楚，**拍板的还是你**。工具再勤快，签字的那支笔，始终在人手里。
