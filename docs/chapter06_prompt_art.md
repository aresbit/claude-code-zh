## 第六章 写提示词的艺术

跟 Claude Code 说话，是门手艺。

它不是人，所以你不能像跟哥们闲扯那样随便；它也不是死机器，所以你不必像写代码那样字字较真。它夹在两者当间，你得寻见那个分寸。

### 一

最要紧的一条：把你要的东西说清楚。

别说「优化这个函数」，太空。是优化性能？优化可读？还是优化错误处理？路子各不相同。

得说明白：

- 要达成什么
- 有什么约束
- 哪些是非做不可，哪些是锦上添花
- 哪些碰不得（比如不许改 API 接口）

这就跟给下属派活一个理儿：交代得越清楚，结果越合心意。

### 二

给它的提示词，最好有个骨架。

我的经验是分作几段：

1. 背景：眼下是个什么情形
2. 目标：要办成什么
3. 约束：有哪些框框
4. 步骤：若有必要，把大致的步子点一点

譬如：

```
背景：我们有一个处理用户登录的函数，眼下没有任何错误处理。
目标：给它加上周全的错误处理。
约束：
- 不许改函数签名
- 不要引入新依赖
- 所有错误都得记日志
步骤：
1. 先把当前代码可能出岔的地方摸一遍
2. 给每个出岔处加上 try-catch
3. 确保每个错误都得到妥善处理
```

这样有骨有架的提示词，它领会起来省力，出的活也更像样。

### 三

还有一招：给例子。

你要是想让它照某种特定的样式或格式出活，最灵的法子就是甩给它一个样品。譬如：

```
请照下面的格式写文档：

## 函数名

简要描述

### 参数
- param1: 描述
- param2: 描述

### 返回值
描述

### 示例
代码示例
```

有了样品，它就知道你要的是哪一路，不至于瞎蒙。

### 四

提示词得一遍遍打磨。

头一回说的，兴许不够清楚。看它的出活，不对，就补料、拨正方向。这是一来一回的对话，不是一锤子的号令。

我惯常的做法是：

1. 先给个大概的提示词
2. 看出活，挑出不顺眼的地方
3. 补说明，正方向
4. 如此反复，直到满意

有时这一来一回得折腾好几趟，可末了的结果，多半不赖。

### 五

末了，记一桩事：Claude Code 有上下文窗口的限。

你给的提示词越长，它能记住的就越少。所以要言简意赅，少说废话。要紧的话搁前头，因为后头的，它兴许就忘了。

这就像给一个记性不好的人写信：要紧事搁开头，且别写太长，不然他读到末尾，就忘了开头讲的什么。

写好提示词，是用 Claude Code 最核心的本事。这要练，要攒经验，要摸熟这只怪兽的脾气。可一旦把这门手艺攥到手里，你就能逼出它最大的劲道。
