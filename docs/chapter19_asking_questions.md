## 第十九章 会问，才有好答案

我年轻时有位老师，特别会问问题。同一个知识点，他问出来的总格外有嚼头，叫人忍不住要去想。后来我才悟到：提问是门手艺，好问题引出好答案，赖问题只换来一句敷衍。

用 Claude Code 也是这样。你怎么问，决定了它怎么答。

### 一

什么是好问题？

头一样，要准。别说「这个不好，改一改」，太空。是功能不对？性能不行？还是有 bug？

好问题是这样的：「这个函数碰上空数组会崩，帮我加道检查。」

第二样，要带上来龙去脉。别当它猜得着你心里想什么，把相干的底细给它：

「我在做一个用户管理系统，用的是 Node.js 和 MongoDB。眼下有个毛病：查用户列表时，数据一多，响应就慢得很。」

### 二

再一样，要分步走。

绕的问题，别指望一口办成。拆成几个小的，一个个来。

譬如要重构一个模块，可以这么走：

1. 先把这个模块眼下的结构捋一捋，主要的毛病在哪儿？
2. 冲着这些毛病，你有什么重构的主意？
3. 就照第二条来，先改接口定义。
4. 这会儿改实现部分。
5. 末了把测试补上。

这样一步一步，比劈头一句「重构这个模块」要稳当得多。

### 三

学会追着问，也要紧。

它的回答，有时不够深，或者没把你的问题答全。这当口要接着追：

- 这一条能不能讲细些？
- 还有别的路子吗？
- 这么做有什么风险？

把问题往深里刨，才挖得出更值钱的东西。

### 四

可也得留神，别一股脑问太多。

上下文窗口有数，问题堆太长，前头的对话就被挤没了；问题一多，它还可能漏掉几条。

我的经验是：一回问三到五个相干的，答完了再问下一拨。这样既不落下，也不撑破上限。

### 五

末了，要学会验。

它给的主意，未必都对——尤其碰上技术的细枝末节，它也会出岔。

所以要紧的东西要核：

- 让它把出处摆出来
- 自己翻文档对一对
- 小范围先试一试

这就像医生开药：药师要审，病人也得瞅瞅说明书，不能闭着眼信。

总之，提问是门手艺。问得在点上，Claude Code 才使得出全力。多练，多归置，你会越来越会跟这只怪兽搭话。
